Create src/utils/marked-config.ts. This utility configures the marked library for the ViteX build pipeline and exports a single `renderMarkdown(content: string): string` function that converts Markdown text to safe HTML.

marked v18 API notes (from STACK.md):
- ESM-only: `import { marked } from 'marked'`
- `.parse()` method accepts a Markdown string and options, returns Promise<string> or string depending on async config
- We use synchronous `.parse(markdown, { async: false })` since we are in a Vite plugin transform hook (sync)
- The `highlight` option was deprecated in marked v5+. For v18, use the `marked-highlight` extension or configure a custom renderer. For this project, use the `marked.use()` API with the `marked-highlight` extension.

Wait -- marked v18 removed the built-in highlight option. The correct approach for marked 18.x is to use the `marked-highlight` extension package. However, that's an additional dependency. Let me use a different approach: configure marked with a custom renderer for code blocks that wraps code in pre/code tags with class names, then apply highlight.js via a post-processing step or via the `marked.use()` extension API.

Actually, the cleanest approach for marked v18:
1. Import `marked` from 'marked'
2. Import `hljs` from 'highlight.js'
3. Use `marked.use({ renderer: { code(...) } })` to override the code renderer
4. The overridden code renderer applies highlight.js to the code content

```typescript
import { marked, type Tokens } from 'marked';
import hljs from 'highlight.js';

// Configure marked for ViteX
marked.use({
  gfm: true,          // GitHub Flavored Markdown (tables, strikethrough, task lists)
  breaks: false,      // Do NOT convert single newlines to <br> (standard Markdown behavior)
});

// Custom renderer for fenced code blocks with syntax highlighting (D-21)
const renderer = {
  code(token: Tokens.Code): string {
    const lang = token.lang || '';
    const code = token.text;

    let highlighted: string;
    if (lang && hljs.getLanguage(lang)) {
      try {
        highlighted = hljs.highlight(code, { language: lang }).value;
      } catch {
        highlighted = escapeHtml(code);
      }
    } else {
      highlighted = escapeHtml(code);
    }

    const langAttr = lang ? ` data-lang="${escapeAttr(lang)}"` : '';
    return `<pre${langAttr}><code class="hljs${lang ? ` language-${lang}` : ''}">${highlighted}</code></pre>`;
  },
};

marked.use({ renderer });

/**
 * Render Markdown content to safe HTML.
 * This runs at build time inside the Vite plugin transform hook.
 * The returned HTML contains no <script>, <iframe>, or event handler attributes.
 */
export function renderMarkdown(content: string): string {
  const html = marked.parse(content, { async: false });
  if (typeof html === 'string') return html;
  // In async mode, marked returns a Promise<string>. With { async: false },
  // it returns string directly. This check satisfies TypeScript.
  throw new Error('marked.parse returned Promise unexpectedly');
}

// Helper: escape HTML special characters (for code content when highlight.js fails)
function escapeHtml(text: string): string {
  return text
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#39;');
}

// Helper: escape attribute values
function escapeAttr(text: string): string {
  return text.replace(/"/g, '&quot;').replace(/</g, '&lt;').replace(/>/g, '&gt;');
}
```

SECURITY NOTES (D-23):
- marked's default behavior in v18 does NOT sanitize HTML -- raw HTML in Markdown is passed through. We need to strip dangerous tags.
- Use the `marked.use()` API to configure a sanitizer.
- marked v18 supports `marked.use({ hooks: { ... } })` for post-processing. However, a simpler approach for our use case (self-authored content, defense in depth) is to use the built-in `sanitizer` option or a custom renderer that only outputs specific tags.
- Since marked v18 removed the `sanitize` option from v0.x days, the recommended approach is: either use DOMPurify (adds dependency) or rely on the fact that all content is self-authored and CSP blocks inline scripts.
- For ViteX: configure marked with `gfm: true` and rely on the Content-Security-Policy meta tag in index.html as defense-in-depth (Plan 01). Also, marked's tokenizer already escapes HTML in most contexts. The CSP `script-src 'self'` blocks any `<script>` tags that might sneak through.
- HOWEVER: to be thorough, add a post-processing step in `renderMarkdown` that strips `<script>`, `<iframe>`, and `on*` attributes using a regex-based sanitizer. Do NOT add DOMPurify as a dependency -- it's 20KB+ and overkill for self-authored content with CSP.

Add this sanitization after marked.parse():

```typescript
export function renderMarkdown(content: string): string {
  let html = marked.parse(content, { async: false });
  if (typeof html !== 'string') {
    throw new Error('marked.parse returned Promise unexpectedly');
  }
  // Strip dangerous tags and attributes (defense in depth, CSP is primary defense)
  html = html.replace(/<script\b[^<]*(?:(?!<\/script>)<[^<]*)*<\/script>/gi, '');
  html = html.replace(/<iframe\b[^<]*(?:(?!<\/iframe>)<[^<]*)*<\/iframe>/gi, '');
  html = html.replace(/\s+on\w+\s*=\s*("[^"]*"|'[^']*'|[^\s>]*)/gi, '');
  return html;
}
```

This regex-based sanitization is sufficient for self-authored content. The CSP meta tag provides the primary XSS defense. The regex strips common injection vectors while preserving all legitimate Markdown-rendered HTML (headings, paragraphs, links, images, code blocks, emphasis).

NOTE: The format/signature of this function is the contract: `renderMarkdown(content: string): string`. The Vite plugin imports and calls this function. It takes raw Markdown and returns safe HTML.

Create vite.config.ts at the project root. This file configures Vite and defines the custom Markdown transform plugin.

The plugin intercepts `.md` file imports in Vite's `transform()` hook (which runs during dev server and production build). For each `.md` file:
1. Parse YAML frontmatter with @11ty/gray-matter (PIPE-02)
2. Render Markdown body to HTML with the renderMarkdown utility (PIPE-03)
3. Export an ES module `{ meta, html }` for consumption by import.meta.glob (PIPE-01)
4. Wrap everything in try/catch -- on failure, log the filename and export a fallback module (PIPE-04)

```typescript
import { defineConfig, type Plugin } from 'vite';
import { resolve } from 'path';
import grayMatter from '@11ty/gray-matter';
import { renderMarkdown } from './src/utils/marked-config';

function markdownPlugin(): Plugin {
  return {
    name: 'vitex-markdown',
    enforce: 'pre', // Run before other plugins

    transform(code: string, id: string) {
      // Only process .md files (PIPE-01: custom Vite plugin discovers .md files)
      if (!id.endsWith('.md')) return;

      // PIPE-04: Per-file try/catch -- one bad file does not crash the entire build
      try {
        // PIPE-02: Parse YAML frontmatter with @11ty/gray-matter
        const { data, content } = grayMatter(code);

        // Extract frontmatter fields, providing defaults for missing data
        const meta = {
          mood: typeof data.mood === 'string' ? data.mood : '📝',
          created: typeof data.created === 'string' ? data.created : new Date(0).toISOString(),
        };

        // PIPE-03: Render Markdown body to HTML at build time
        const html = renderMarkdown(content.trim());

        // Export as ES module -- shape must match src/types/markdown.d.ts
        return {
          code: [
            `export const meta = ${JSON.stringify(meta)};`,
            `export const html = ${JSON.stringify(html)};`,
          ].join('\n'),
          map: null,
        };
      } catch (err: unknown) {
        // PIPE-04: Log the filename and skip broken files
        const message = err instanceof Error ? err.message : String(err);
        console.warn(`[vitex-markdown] Skipping ${id}: ${message}`);

        // Export a fallback module so the build continues
        return {
          code: [
            `export const meta = { mood: "⚠️", created: "${new Date(0).toISOString()}" };`,
            `export const html = "<p><em>This tweet could not be parsed.</em></p>";`,
          ].join('\n'),
          map: null,
        };
      }
    },

    // Handle HMR for .md files: when a Markdown file changes, do a full page reload.
    // This is necessary because .md files are processed at the module level --
    // changing one tweet means the glob-constructed tweets array needs to be rebuilt.
    handleHotUpdate({ file, server }) {
      if (file.endsWith('.md')) {
        console.log(`[vitex-markdown] Markdown file changed: ${file}`);
        // Full reload is the safest for content changes since
        // import.meta.glob with eager:true caches the entire collection
        server.ws.send({ type: 'full-reload' });
      }
    },
  };
}

export default defineConfig({
  plugins: [markdownPlugin()],

  resolve: {
    alias: {
      '@': resolve(__dirname, 'src'),
    },
  },

  // Ensure .md files are treated as modules (not static assets)
  assetsInclude: [],

  build: {
    target: 'es2022',
    outDir: 'dist',
    emptyOutDir: true,
    rollupOptions: {
      input: resolve(__dirname, 'index.html'),
    },
  },

  server: {
    port: 3000,
    open: true,
  },
});
```

Key design decisions:
- `enforce: 'pre'` ensures the Markdown plugin runs before Vite's built-in asset handling. This prevents `.md` files from being treated as static text assets.
- The module export shape (`{ meta, html }`) MUST match exactly what `src/types/markdown.d.ts` declares and what `src/data/tweets.ts` (Plan 04) expects from `import.meta.glob`.
- `meta.created` defaults to Unix epoch zero (`1970-01-01T00:00:00.000Z`) if the frontmatter is missing a `created` field. This ensures the tweet sorts to the bottom of the timeline rather than crashing.
- `meta.mood` defaults to `📝` if missing -- a neutral placeholder emoji.
- The error fallback module exports a warning emoji (`⚠️`) and a placeholder message so the timeline still renders, just with a visibly broken card that the author can identify.
- `handleHotUpdate` triggers a full page reload on `.md` file changes. This is intentional: `import.meta.glob` with `eager: true` bundles all content into a single array at module init time. A single-file HMR update would not refresh the glob collection. Full reload ensures the updated tweet appears correctly.
- Server port 3000 (can be changed, but 3000 is the standard dev port).
- `build.target: 'es2022'` for modern browsers with full Intl.RelativeTimeFormat support (needed by dayjs in Plan 04).
- No additional plugins needed -- the Markdown plugin is the only one.

Do NOT: add PostCSS, Tailwind, or any CSS processing (Pico.css is imported directly in styles.css). Do NOT add react, vue, or svelte plugins.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Markdown files (content/tweets/) -> Vite plugin | User-authored Markdown content enters the build pipeline. Malformed YAML or malicious Markdown could disrupt the build or produce unsafe HTML. |
| Vite plugin output -> Browser bundle | The plugin's HTML output is embedded in the JS bundle and served to the browser. Unsanitized HTML is an XSS vector. |
| highlight.js -> Browser bundle | highlight.js is a build-time dependency but adds CSS classes to code blocks. Browser applies styles from styles.css only -- no highlight.js JS is shipped to the client. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Markdown files | mitigate | Malformed YAML does not crash the build -- per-file try/catch logs the filename and exports a fallback module. The site deploys with the broken tweet replaced by a warning placeholder. Other tweets render normally. |
| T-03-02 | Elevation of Privilege | marked HTML output | mitigate | renderMarkdown() strips <script> tags, <iframe> tags, and on* event handler attributes from the HTML output. Content-Security-Policy meta tag in index.html provides defense-in-depth by blocking script execution from inline or external sources. |
| T-03-03 | Information Disclosure | Frontmatter values | mitigate | Only explicitly extracted frontmatter fields (mood, created) are exported to the client bundle. Any additional frontmatter keys the user adds are silently ignored by the plugin, not leaked to the browser. |
| T-03-04 | Denial of Service | Build pipeline | accept | If highlight.js throws on malformed code input, the renderer falls back to escapeHtml(). If gray-matter throws on malformed YAML, the per-file catch block handles it. No single file can crash the entire build. Build timeout from extremely large content volumes is addressed in Phase 3. |
</threat_model>
